# 动态数据存储建议 (Dynamic Data Storage Guide)

针对水库知识图谱中的“动态数据”和“时序数据”（例如：实时水位、小时级流量、历史灾害事件），**强烈不建议直接存入 Neo4j 作为无属性图节点爆炸式增长**。

以下是主流生产环境推荐的混合存储架构：

## 1. Neo4j：只存“位置、关系、元数据”
知识图谱（Neo4j）的优势在于处理拓扑关系，而非计算和存储海量时间点数据。
*   **合适存的**：水文站名称、经纬度、水文站属于哪个水库、预警阈值（汛限水位静态值）。
*   **不该存的**：每天早中晚各测一次的具体水位数值（一年有成千上万条记录，在图谱里会导致严重的“超级节点”即 Super Node 图遍历性能灾难）。

## 2. 关系型数据库 (MySQL/PostgreSQL) 或时序数据库 (InfluxDB/TDengine)：存“数字指标流水”
真正的时序数据，必须交给专业的数据库或传统 RDBMS 处理。
*   **时序数据库 (InfluxDB 等)**：适用于极高频采样（例如每 5 分钟汇报一次传感器指标）。
*   **关系型数据库 (MySQL)**：如果你其实只有“每日水位”、“每日入库流量”等低频或中频数据，用你现在的 MySQL（在 `main.py` 中看到了你使用 `aiomysql` 和 SQLAlchemy）完全足够了。

### ✅ 混合架构设计方案（Graph + RDBMS）

你的架构应该是：
1. **Neo4j（图谱）负责语义导航：**
   用户问：*“丹江口水库上游的所有水文站，当前水位情况如何？”*
   系统向 Neo4j 请求：获取 `丹江口水库` 经由 `UPSTREAM_OF` 关联的所有 `水文站` 的 ID 和编号。
2. **MySQL（关系型/时序记录）负责精准查询：**
   拿到上一阶段这批水文站的 ID 后，系统拿着 ID 数组到 MySQL 库里的 `water_level_sensor_log` 表中，查询最近 24 小时的平均水位或最新记录。

---

## 3. 在 FastAPI 中如何落地这个混合架构？

你要做的事情是引入 **Function Calling (大模型工具调用)** 或者将你 `qa_service.py` 里的规则流水线增加一个“动态特征分支”。

例如，当检测到关键词含有 `"当前水位"`、`"今天"`、`"实时"` 时：
1. 还是执行 Neo4j，查到对应的目标节点标识。
2. 在 FastAPI 中单独加一个内部函数：`get_real_time_water_level(reservoir_id)`。
3. 从 MySQL 中读取该 ID 对应的最近记录并组装为一段回复，返回给用户。

如果你后期真的有了这类动态传感器或者日报表数据，可以直接在你的 `main.py` 的 SQLAlchemy 连接中起一张 `WaterLevelRecord` 表，然后在大模型返回链路里插一环即可。
